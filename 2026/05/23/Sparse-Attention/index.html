稀疏注意力

Created2026-05-23|Updated2026-05-24|大模型优化
|Word Count:6.7k|Reading Time:26mins|Post Views:

前言

基础的dense attention的计算复杂度为O(N2d)O(N^2d)O(N​2​​d)，当上下文很长时，这个计算会变得非常复杂，所以需要稀疏注意力以减小计算复杂度。

基本思想为，放弃计算每两个token之间的attention分数，优化为选取部分token计算注意力分数，在大幅度降低计算复杂度的同时使模型能学会全局特征

OpenAI 稀疏注意力

步长稀疏注意力 (Strided Attention)

假设我们有一个由 16 个 Token 组成的超短序列（T1T_1T​1​​ 到 T16T_{16}T​16​​）。
我们要应用步长稀疏注意力 (Strided Attention)，并且在模型里做如下两个超参数的配置：

  1. 步长 (Stride) S=4S = 4S=4
  2. 方向：为了简单直观，我们先不考虑自回归的单向掩码，假设这是一个**双向（Bidirectional）**注意力（即后面的 Token 也可以看前面的 Token）。

根据 OpenAI 的定义，在这种配置下，任意一个 Token TiT_iT​i​​ 在 Token 级别只能注意到两类位置：

  • 最近邻居（Local 视图）：距离自己不超过步长 SSS 的连续 Token。
  • 周期跨步（Strided 视图）：所有索引能被 SSS 整除（或者是其余数对齐）的跳跃 Token。

我们站在 T9T_9T​9​​ 的视角，看看它在计算自注意力时，到底会和序列中的哪些 Token 发生“点积”计算。

  1. Local 视图（看身边）：

    👉 激活位置：T7,T8,T9,T10,T11T_7, T_8, T_9, T_{10}, T_{11}T​7​​,T​8​​,T​9​​,T​10​​,T​11​​（通常在切块边界内取近邻，这里覆盖了它最亲密的上下文）。
  2. Strided 视图（看远方）：
    T9T_9T​9​​ 会以步长 S=4S=4S=4 向前、向后发射“探测波”，只看那些和自己处于相同周期相位的位置（即索引满足 9±4×k9 \pm 4 \times k9±4×k 的位置）。
    👉 激活位置：T1,T5,T13T_1, T_5, T_{13}T​1​​,T​5​​,T​13​​ （9−4=59-4=59−4=5, 5−4=15-4=15−4=1, 9+4=139+4=139+4=13）。

S9={T1,T5,T7,T8,T9,T10,T11,T13}S_9 = \{T_1, T_5, T_7, T_8, T_9, T_{10}, T_{11}, T_{13}\} S​9​​={T​1​​,T​5​​,T​7​​,T​8​​,T​9​​,T​10​​,T​11​​,T​13​​}

原本在 Dense Attention 中，T9T_9T​9​​ 需要和全部 16 个 Token 计算关系；而现在，它只需要和这 8 个 Token 计算，计算量直接砍掉了一半。

💡 为什么这种“跳着算”的结构能work？

你可能会问：T9T_9T​9​​连 T2,T3,T4T_2, T_3, T_4T​2​​,T​3​​,T​4​​ 都不看，信息不就丢了吗？

这就是 OpenAI 这篇设计最精妙的地方——多层网络的信息复用（Information Routing）。

  • 在第 1 层网络中：
    T5T_5T​5​​ 因为 Local 视图，看全了 T2,T3,T4,T5,T6T_2, T_3, T_4, T_5, T_6T​2​​,T​3​​,T​4​​,T​5​​,T​6​​。所以计算结束后，T5T_5T​5​​ 的输出向量里已经融合了 T2,T3,T4T_2, T_3, T_4T​2​​,T​3​​,T​4​​ 的信息。
  • 到了第 2 层网络中：
    T9T_9T​9​​ 通过 Strided 视图去看了 T5T_5T​5​​。因为此时的 T5T_5T​5​​ 已经不是纯粹的 T5T_5T​5​​ 了，它是一个“聚合了第1层 2、3、4号Token”的中间体。

通过这种**“第1层隔空收集，第2层二手中转”**的拓扑结构，整个长文本的信息流在短短 2 层 Layer 内，就完成了全局的互通。既保证了感受野（Receptive Field）覆盖到长文本的每一个角落，又在 Token 级别省下了大量的无效点积计算。

固定稀疏注意力 (Fixed Attention)

固定稀疏注意力（Fixed Attention）在设计上更加具有针对性，尤其是为了文本（Text）这种没有固定周期、但存在长距离语义关联的数据而设计的。

它的核心思想是：既然文本中有些词非常重要（比如句首的特殊 Token，或者段落的总结词），那么我们就人为在序列中指定一些**“全局中转站（Global Aggregators）”**。所有的 Token 都可以和这些中转站通信，从而实现全局信息的流转。

为了和前面的步长注意力作对比，我们同样用一个长度 N=16N = 16N=16 的序列（T1T_1T​1​​ 到 T16T_{16}T​16​​）来举例。
我们设定：

  1. 块大小 / 步长 S=4S = 4S=4
  2. 全局中转站的设定规则：每个块（Block）的最后一个 Token 被指定为全局中转站（即 T4,T8,T12,T16T_4, T_8, T_{12}, T_{16}T​4​​,T​8​​,T​12​​,T​16​​）。

🔍 同样以第 9 个 Token（T9T_9T​9​​）为例

在固定稀疏注意力下，任意一个 Token TiT_iT​i​​ 只能关注：

  • 局部邻居（Local 视图）：自己所在块（Block）内部的所有 Token。
  • 固定中转站（Global 视图）：序列中所有被指定为中转站的 Token。

我们来看看 T9T_9T​9​​ 的视角：

第一步：找出 T9T_9T​9​​ 允许关注的 Token 集合 S9S_9S​9​​

  1. Local 视图（看内部）：
    T9T_9T​9​​ 属于第三个块（包含 T9,T10,T11,T12T_9, T_{10}, T_{11}, T_{12}T​9​​,T​10​​,T​11​​,T​12​​）。在这个块内，它会和所有邻居打招呼。
    👉 激活位置：T9,T10,T11,T12T_9, T_{10}, T_{11}, T_{12}T​9​​,T​10​​,T​11​​,T​12​​。
  2. Global 视图（看中转站）：
    T9T_9T​9​​ 获准去链接全序列所有的“中转站” Token。
    👉 激活位置：T4,T8,T12,T16T_4, T_8, T_{12}, T_{16}T​4​​,T​8​​,T​12​​,T​16​​。

将两者合并，T9T_9T​9​​ 最终的注意力集合为：

S9={T4,T8,T9,T10,T11,T12,T16}S_9 = \{T_4, T_8, T_9, T_{10}, T_{11}, T_{12}, T_{16}\} S​9​​={T​4​​,T​8​​,T​9​​,T​10​​,T​11​​,T​12​​,T​16​​}

此时，T9T_9T​9​​ 只需要和这 7 个 Token 进行点积计算（重叠的 T12T_{12}T​12​​ 只算一次）。

🔄 换个视角：如果是中转站 Token 呢？

为了让这个网络运转起来，中转站 Token（比如 T8T_8T​8​​）的权力极大：
根据规则，T8T_8T​8​​ 的 Local 视图是它所在的第二块（T5,T6,T7,T8T_5, T_6, T_7, T_8T​5​​,T​6​​,T​7​​,T​8​​），而它的 Global 视图允许它看所有的中转站。
但更关键的是，所有其他的普通 Token 在计算时都在看着 T8T_8T​8​​。

这就导致在矩阵乘法的反向传播和信息流中：

  • T8T_8T​8​​ 收集了它所在局部块（T_5 \dots T_8）的所有核心语义。
  • 当 T9T_9T​9​​ 在计算时，它直接去看了 T8T_8T​8​​。
  • 👉 结果：T9T_9T​9​​ 仅仅通过和 T8T_8T​8​​ 计算了一次点积，就间接获取了上一个块中 T5,T6,T7T_5, T_6, T_7T​5​​,T​6​​,T​7​​ 的所有合并信息。

⚖️ 步长 (Strided) vs 固定 (Fixed) 有什么本质区别？

我们可以用一个通俗的比喻来总结 OpenAI 的这两个设计：

  • 步长稀疏注意力 (Strided) 像是一个**“巡逻兵”。每个人都按照固定的步伐（比如每隔 4 步）向后看。它非常死板，但在处理图像（图像有规律的行高、列宽）或音频（固定的采样率）这种空间/时间上具有严格周期性**的数据时，极其高效。
  • 固定稀疏注意力 (Fixed) 像是一个**“客栈/邮局”**。每隔一段距离建一个邮局（T_4, T_8 \dots）。普通人有事就去最近的邮局存取信件，邮局之间互通有无。文本数据没有死板的周期性，但通过这种“邮局中转”的拓扑架构，能用极低的计算代价实现跨越数万 token 的全局宏观对话。

DeepSpeed Sparse Attention – 块稀疏（Block-Sparse）

主要是引入了“块稀疏（Block-Sparse）”的概念

块稀疏（Block-Sparse）

之前的做法都是以token为基础单位，但是如果在 GPU 里直接做元素级（Element-wise）的离散筛选（比如让 T9T_9T​9​​ 只算 T1,T5,T7T_1, T_5, T_7T​1​​,T​5​​,T​7​​），会导致大量的碎片化内存访问（Strided Memory Access）。GPU 的张量核心（Tensor Cores）根本跑不起来，反而比全连接还要慢。

为了迎合 GPU 的合并内存访问特性，DeepSpeed 引入了块稀疏（Block-Sparse）：

  • 切块（Blocking）：把整个 N×NN \times NN×N 的大矩阵切成一个个固定大小的方块（Block，通常是 16×1616 \times 1616×16 或 64×6464 \times 6464×64）。
  • 块级筛选（Block Mask）：在块的粒度上定义哪些块需要计算（设为 1），哪些块直接跳过（设为 0）。只要被激活，块内部的 Token 依然进行密集、高效的矩阵乘法。

举例

假设现在计算第100个token的注意力权重，通过算法，发现其需要与第200个和第400个token进行计算

🛠️ 硬件底层的真实计算流水线

假设你的 Block Size 设为 161616（也就是说，每 16 个 Token 打包成一个块）。

  • 第 200 个 Token 落在第 13 个块（200÷16=12.5200 \div 16 = 12.5200÷16=12.5）。
  • 第 400 个 Token 落在第 25 个块（400÷16=25400 \div 16 = 25400÷16=25）。（这里借用你的设定，我们直接称它们为块 A 和块 B）。当模型处理到第 100 个 Token（所在的块称为 Query 块）时，硬件（Triton 算子）是这样暴力而又高效地执行的：

第二步：Token 级别的密集计算（Intra-block Dense Dot-Product）

数据进到 SRAM 后，硬件会直接利用 Tensor Cores 跑最擅长的矩阵乘法。

  • 它让第 100 个 Token 的 Query，与块 A 里的全部 16 个 Key、以及块 B 里的全部 16 个 Key 统一算一遍点积。
  • 这里产生了一个工程细节：虽然你的算法逻辑上可能只需要第 200 个和第 400 个 Token，但因为整块都被搬进来了，硬件在计算时，实际上把这两个块里的 32 个 Token 全都顺便算了一遍点积。
  • 为什么这样做反而快？ 因为对于 GPU 而言，连续算 32 个点积，比它在海量显存里“挑挑拣拣”只算 2 个点积，速度要快上成百上千倍。

第三步：掩码过滤与归一化（Masking & Softmax）

算完这 32 个点积得分后，DeepSpeed 会在底层套上你最初设定好的算法掩码（Mask）：

  • 那些被顺便算出来的、算法不想要的 Token（比如第 199 个、第 401 个 Token），它们的点积得分会被无情地加上 −∞-\infty−∞。
  • 经过 Softmax 之后，这些不需要的 Token 的权重严格归零。
  • 最终，只有第 200 个和第 400 个 Token 获得了真正的注意力权重（不为 0），并去抽取对应的 Value 向量。

总结

  • 在显存和算力开销上：它付出的代价是“块”级别的。没被选中的那几百个块，GPU 连看都不看，直接跳过，从而省下了 90% 以上的显存和算力。
  • 在局部计算上：一旦某个块被选中，它在内部付出的代价是“密集点积”的。硬件把整块拿出来算，再通过 Mask 在 Token 级别筛选出你真正想要的第 200 和 400 个 Token。

Native Sparse Attention

1、粗粒度压缩块（Coarse-grained Compressed Blocks）

中心思想：即将N个token的key压缩为n个子key，其中N>>n，可以简单理解为将多个token的key送入MLP或卷积神经网络输出单token长度的key

论文中关于粗粒度压缩块的 Key 集合定义为：

\tilde{K}_t^{\text{cmp}} = f_K^{\text{cmp}}(\mathbf{k}_{:t}) = \left\{ \varphi(\mathbf{k}_{id+1:id+l}) \;\middle|\; 0 \le i \le \left\lfloor \frac{t - l}{d} \right\rfloor \right\}

各个变量、下标及其对应的物理含义如下：

  • ttt：当前正在处理的 Token 级别的时间步（序列长度）。
  • k:t\mathbf{k}_{:t}k​:t​​：到当前时间步为止，全量、连续的原始 Key 向量序列。
  • lll：压缩窗口的长度（Window Length）。也就是每次拿多少个连续的原始 Token 融合成一个超级 Token。
  • ddd：滑动步长（Stride）。即压缩窗口每次向前移动多少个 Token。
  • φ\varphiφ：压缩函数（Compression Function）。负责把输入的 lll 个原始 Token 向量变成一个压缩后的 Token 向量（如平均池化或线性投影）。
  • iii：压缩后生成的新序列的 Token 索引。

核心机制：自适应滑动压缩（Sliding/Strided Compression）

根据公式，DeepSeek 在这里实现的并不是死板的、互不重叠的“块内池化”，而是一个带有步长 ddd 的滑动窗口压缩机制：

  1. 窗口切片映射：kid+1:id+l\mathbf{k}_{id+1:id+l}k​id+1:id+l​​
  • 当 i=0i=0i=0 时，计算第一个压缩 Token，它调阅原始序列的 k1:l\mathbf{k}_{1:l}k​1:l​​（第 111 到第 lll 个 Token）。
  • 当 i=1i=1i=1 时，计算第二个压缩 Token，窗口向前移动了 ddd 个位置，调阅原始序列的 kd+1:d+l\mathbf{k}_{d+1:d+l}k​d+1:d+l​​。
  • 重叠边界（Overlap）：若 d<ld < ld<l，意味着前后相邻的压缩块之间存在重叠。这种设计能够极大地保留上下文在分块边界处的连续性，有效防止硬切块带来的语义断层。
  1. 终止边界控制：⌊t−ld⌋\left\lfloor \frac{t - l}{d} \right\rfloor⌊​d​​t−l​​⌋
    这个下取整的上限精确控制了压缩窗口最远能滑到哪里：

2、粗粒度快与细粒度块的转换

这里回答了一个问题，之前计算的是粗粒度块的信息，但是在后续计算时需要使用细粒度块的数据，如何转换？

在 NSA 的设计中，压缩分支和选择分支各过各的，它们的方块大小和滑动方式是不一样的：

  • 粗粒度压缩分支：由于要滑得很密集，它的步长设得比较小（比如 d=8d=8d=8）。意味着在显存里，每隔 8 个 Token 就有一个压缩代表。
  • 细粒度选择分支：由于要把原始 Token 整个搬进 SRAM，为了符合 GPU 的计算吞吐，它划分的密集块通常比较大（比如 l′=32l'=32l​′​​=32），且通常不重叠。

因为网格大小不一样，你没法直接连线。你必须把物理空间上“落在这个 32 维大块内部”的所有小压缩 Token 的得分给挑出来，全部加在一起。

3、多头注意力得分统一

如果每个头得到的Top-K个细粒度块都不一样，假设有8个head，因为这 8 个 Head 共享同一份 KV Cache 内存。当 GPU 的线程束（Warps）准备去显存里搬运数据时，它发现这 8 个 Head 要去 8 个完全不连续的、甚至隔了十万八千里的内存地址（第 3、50、99 块）去读数据。

这在显存里叫非合并访问（Uncoalesced Memory Access）/ 内存分叉（Memory Divergence）。
GPU 无法一次性大批量把数据运回来，只能像送外卖一样，派 8 个小哥分别去 8 个地方单点取货。稀疏注意力本来是为了省带宽、提速，结果多头的意见不合，反而把 GPU 堵死在频繁读取非连续显存的延迟（Latency）里了。
