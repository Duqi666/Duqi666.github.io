稀疏注意力

Created2026-05-23|Updated2026-05-24|大模型优化
|Word Count:5.8k|Reading Time:20mins|Post Views:

前言

基础的dense attention的计算复杂度为$O(N^2d)$，当上下文很长时，这个计算会变得非常复杂，所以需要稀疏注意力以减小计算复杂度。

基本思想为，放弃计算每两个token之间的attention分数，优化为选取部分token计算注意力分数，在大幅度降低计算复杂度的同时使模型能学会全局特征

OpenAI 稀疏注意力

步长稀疏注意力 (Strided Attention)

假设我们有一个由 16 个 Token 组成的超短序列（$T1$ 到 $T{16}$）。
我们要应用步长稀疏注意力 (Strided Attention)，并且在模型里做如下两个超参数的配置：

  1. 步长 (Stride) $S = 4$
  2. 方向：为了简单直观，我们先不考虑自回归的单向掩码，假设这是一个双向（Bidirectional）注意力（即后面的 Token 也可以看前面的 Token）。

根据 OpenAI 的定义，在这种配置下，任意一个 Token $T_i$ 在 Token 级别只能注意到两类位置：

  • 最近邻居（Local 视图）：距离自己不超过步长 $S$ 的连续 Token。
  • 周期跨步（Strided 视图）：所有索引能被 $S$ 整除（或者是其余数对齐）的跳跃 Token。

🔍 以第 9 个 Token（$T_9$）为例

我们站在 $T_9$ 的视角，看看它在计算自注意力时，到底会和序列中的哪些 Token 发生“点积”计算。

第一步：找出 $T_9$ 允许关注的 Token 集合 $S_9$

  1. Local 视图（看身边）：

💡 为什么这种“跳着算”的结构能work？

你可能会问：$T_9$连 $T_2, T_3, T_4$ 都不看，信息不就丢了吗？

这就是 OpenAI 这篇设计最精妙的地方——多层网络的信息复用（Information Routing）。

  • 在第 1 层网络中：
    $T_5$ 因为 Local 视图，看全了 $T_2, T_3, T_4, T_5, T_6$。所以计算结束后，$T_5$ 的输出向量里已经融合了 $T_2, T_3, T_4$ 的信息。
  • 到了第 2 层网络中：
    $T_9$ 通过 Strided 视图去看了 $T_5$。因为此时的 $T_5$ 已经不是纯粹的 $T_5$ 了，它是一个“聚合了第1层 2、3、4号Token”的中间体。

通过这种“第1层隔空收集，第2层二手中转”的拓扑结构，整个长文本的信息流在短短 2 层 Layer 内，就完成了全局的互通。既保证了感受野（Receptive Field）覆盖到长文本的每一个角落，又在 Token 级别省下了大量的无效点积计算。

固定稀疏注意力 (Fixed Attention)

固定稀疏注意力（Fixed Attention）在设计上更加具有针对性，尤其是为了文本（Text）这种没有固定周期、但存在长距离语义关联的数据而设计的。

它的核心思想是：既然文本中有些词非常重要（比如句首的特殊 Token，或者段落的总结词），那么我们就人为在序列中指定一些“全局中转站（Global Aggregators）”。所有的 Token 都可以和这些中转站通信，从而实现全局信息的流转。

为了和前面的步长注意力作对比，我们同样用一个长度 $N = 16$ 的序列（$T1$ 到 $T{16}$）来举例。
我们设定：

  1. 块大小 / 步长 $S = 4$
  2. 全局中转站的设定规则：每个块（Block）的最后一个 Token 被指定为全局中转站（即 $T4, T_8, T{12}, T_{16}$）。

🔍 同样以第 9 个 Token（$T_9$）为例

在固定稀疏注意力下，任意一个 Token $T_i$ 只能关注：

  • 局部邻居（Local 视图）：自己所在块（Block）内部的所有 Token。
  • 固定中转站（Global 视图）：序列中所有被指定为中转站的 Token。

我们来看看 $T_9$ 的视角：

第一步：找出 $T_9$ 允许关注的 Token 集合 $S_9$

  1. Local 视图（看内部）：
    $T9$ 属于第三个块（包含 $T_9, T{10}, T{11}, T{12}$）。在这个块内，它会和所有邻居打招呼。
    👉 激活位置：$T9, T{10}, T{11}, T{12}$。
  2. Global 视图（看中转站）：
    $T9$ 获准去链接全序列所有的“中转站” Token。
    👉 激活位置：**$T_4, T_8, T    {12}, T_{16}$**。

🔄 换个视角：如果是中转站 Token 呢？

为了让这个网络运转起来，中转站 Token（比如 $T_8$）的权力极大：
根据规则，$T_8$ 的 Local 视图是它所在的第二块（$T_5, T_6, T_7, T_8$），而它的 Global 视图允许它看所有的中转站。
但更关键的是，所有其他的普通 Token 在计算时都在看着 $T_8$。

这就导致在矩阵乘法的反向传播和信息流中：

  • $T_8$ 收集了它所在局部块（$T_5 \dots T_8$）的所有核心语义。
  • 当 $T_9$ 在计算时，它直接去看了 $T_8$。
  • 👉 结果：$T_9$ 仅仅通过和 $T_8$ 计算了一次点积，就间接获取了上一个块中 $T_5, T_6, T_7$ 的所有合并信息。

⚖️ 步长 (Strided) vs 固定 (Fixed) 有什么本质区别？

我们可以用一个通俗的比喻来总结 OpenAI 的这两个设计：

  • 步长稀疏注意力 (Strided) 像是一个“巡逻兵”。每个人都按照固定的步伐（比如每隔 4 步）向后看。它非常死板，但在处理图像（图像有规律的行高、列宽）或音频（固定的采样率）这种空间/时间上具有严格周期性的数据时，极其高效。
  • 固定稀疏注意力 (Fixed) 像是一个“客栈/邮局”。每隔一段距离建一个邮局（$T_4, T_8 \dots$）。普通人有事就去最近的邮局存取信件，邮局之间互通有无。文本数据没有死板的周期性，但通过这种“邮局中转”的拓扑架构，能用极低的计算代价实现跨越数万 token 的全局宏观对话。

DeepSpeed Sparse Attention — 块稀疏（Block-Sparse）

主要是引入了“块稀疏（Block-Sparse）”的概念

块稀疏（Block-Sparse）

之前的做法都是以token为基础单位，但是如果在 GPU 里直接做元素级（Element-wise）的离散筛选（比如让 $T_9$ 只算 $T_1, T_5, T_7$），会导致大量的碎片化内存访问（Strided Memory Access）。GPU 的张量核心（Tensor Cores）根本跑不起来，反而比全连接还要慢。

为了迎合 GPU 的合并内存访问特性，DeepSpeed 引入了块稀疏（Block-Sparse）：

  • 切块（Blocking）：把整个 $N \times N$ 的大矩阵切成一个个固定大小的方块（Block，通常是 $16 \times 16$ 或 $64 \times 64$）。
  • 块级筛选（Block Mask）：在块的粒度上定义哪些块需要计算（设为 1），哪些块直接跳过（设为 0）。只要被激活，块内部的 Token 依然进行密集、高效的矩阵乘法。

举例

假设现在计算第100个token的注意力权重，通过算法，发现其需要与第200个和第400个token进行计算

🛠️ 硬件底层的真实计算流水线

假设你的 Block Size 设为 $16$（也就是说，每 16 个 Token 打包成一个块）。

  • 第 200 个 Token 落在第 13 个块（$200 \div 16 = 12.5$）。
  • 第 400 个 Token 落在第 25 个块（$400 \div 16 = 25$）。（这里借用你的设定，我们直接称它们为块 A 和块 B）。当模型处理到第 100 个 Token（所在的块称为 Query 块）时，硬件（Triton 算子）是这样暴力而又高效地执行的：

第二步：Token 级别的密集计算（Intra-block Dense Dot-Product）

数据进到 SRAM 后，硬件会直接利用 Tensor Cores 跑最擅长的矩阵乘法。

  • 它让第 100 个 Token 的 Query，与块 A 里的全部 16 个 Key、以及块 B 里的全部 16 个 Key 统一算一遍点积。
  • 这里产生了一个工程细节：虽然你的算法逻辑上可能只需要第 200 个和第 400 个 Token，但因为整块都被搬进来了，硬件在计算时，实际上把这两个块里的 32 个 Token 全都顺便算了一遍点积。
  • 为什么这样做反而快？ 因为对于 GPU 而言，连续算 32 个点积，比它在海量显存里“挑挑拣拣”只算 2 个点积，速度要快上成百上千倍。

第三步：掩码过滤与归一化（Masking & Softmax）

算完这 32 个点积得分后，DeepSpeed 会在底层套上你最初设定好的算法掩码（Mask）：

  • 那些被顺便算出来的、算法不想要的 Token（比如第 199 个、第 401 个 Token），它们的点积得分会被无情地加上 $-\infty$。
  • 经过 Softmax 之后，这些不需要的 Token 的权重严格归零。
  • 最终，只有第 200 个和第 400 个 Token 获得了真正的注意力权重（不为 0），并去抽取对应的 Value 向量。

总结

  • 在显存和算力开销上：它付出的代价是“块”级别的。没被选中的那几百个块，GPU 连看都不看，直接跳过，从而省下了 90% 以上的显存和算力。
  • 在局部计算上：一旦某个块被选中，它在内部付出的代价是“密集点积”的。硬件把整块拿出来算，再通过 Mask 在 Token 级别筛选出你真正想要的第 200 和 400 个 Token。

    Native Sparse Attention

1、粗粒度压缩块（Coarse-grained Compressed Blocks）

中心思想：即将N个token的key压缩为n个子key，其中N>>n，可以简单理解为将多个token的key送入MLP或卷积神经网络输出单token长度的key

论文中关于粗粒度压缩块的 Key 集合定义为：

各个变量、下标及其对应的物理含义如下：

  • $t$：当前正在处理的 Token 级别的时间步（序列长度）。
  • $\mathbf{k}_{:t}$：到当前时间步为止，全量、连续的原始 Key 向量序列。
  • $l$：压缩窗口的长度（Window Length）。也就是每次拿多少个连续的原始 Token 融合成一个超级 Token。
  • $d$：滑动步长（Stride）。即压缩窗口每次向前移动多少个 Token。
  • $\varphi$：压缩函数（Compression Function）。负责把输入的 $l$ 个原始 Token 向量变成一个压缩后的 Token 向量（如平均池化或线性投影）。
  • $i$：压缩后生成的新序列的 Token 索引。

核心机制：自适应滑动压缩（Sliding/Strided Compression）

根据公式，DeepSeek 在这里实现的并不是死板的、互不重叠的“块内池化”，而是一个带有步长 $d$ 的滑动窗口压缩机制：

  1. 窗口切片映射：$\mathbf{k}_{id+1:id+l}$
  • 当 $i=0$ 时，计算第一个压缩 Token，它调阅原始序列的 $\mathbf{k}_{1:l}$（第 $1$ 到第 $l$ 个 Token）。
  • 当 $i=1$ 时，计算第二个压缩 Token，窗口向前移动了 $d$ 个位置，调阅原始序列的 $\mathbf{k}_{d+1:d+l}$。
  • 重叠边界（Overlap）：若 $d < l$，意味着前后相邻的压缩块之间存在重叠。这种设计能够极大地保留上下文在分块边界处的连续性，有效防止硬切块带来的语义断层。
  1. 终止边界控制：$\left\lfloor \frac{t - l}{d} \right\rfloor$
    这个下取整的上限精确控制了压缩窗口最远能滑到哪里：
  • 随着窗口不断右移，其右边界（$id+l$）绝对不会超过或逼近当前的 $t$。
  • 该边界在远端压缩历史与近端局部窗口（Sliding Window）之间划清了界限，确保被压缩的全部都是历史深处的陈旧数据，而最新的临近数据绝对不参与任何模糊化压缩。

工程应用实例

假设当前处理到 $t=1000$ 个 Token，设定压缩窗长 $l=16$，滑动步长 $d=8$（即保持 50% 的重叠度）：

  • 当 $i=0$ 时，$\varphi$ 将 $\mathbf{k}_{1:16}$ 压缩成 $\tilde{K}_t^{\text{cmp}}[0]$。
  • 当 $i=1$ 时，$\varphi$ 将 $\mathbf{k}_{9:24}$ 压缩成 $\tilde{K}_t^{\text{cmp}}[1]$。
  • 依此类推，窗口持续右移，直到上限 $i = \lfloor (1000-16)/8 \rfloor = 123$。

最终，原本前 1000 个复杂的历史 Token，在硬件执行层面整齐地缩减为了由 124 个压缩 Token 构成的连续紧凑矩阵 $\tilde{K}_t^{\text{cmp}}$，当前的 Query 只需与这 124 个高级特征进行全量点积，大幅拉升了计算速度与显存带宽效率。

2、Token Selection

中心思想：对于某个token，选择其最相关的p个细粒度块并计算attention。在第一步中，N个token的key被压缩到了n维，所以$\tilde{K}_t^{\text{cmp}}$的大小应该是$n×d$

1、计算token与粗粒度块的注意力得分

表示第t个token与所有粗粒度块的注意力得分，所以$\mathbf{p}_t^{\text{cmp}}$的大小为$1×n$，那么对于T个token，会得到一个大小为$T×n$的注意力矩阵

2、粗粒度快与细粒度块的转换

这里回答了一个问题，之前计算的是粗粒度块的信息，但是在后续计算时需要使用细粒度块的数据，如何转换？

在 NSA 的设计中，压缩分支和选择分支各过各的，它们的方块大小和滑动方式是不一样的：

  • 粗粒度压缩分支：由于要滑得很密集，它的步长设得比较小（比如 $d=8$）。意味着在显存里，每隔 8 个 Token 就有一个压缩代表。
  • 细粒度选择分支：由于要把原始 Token 整个搬进 SRAM，为了符合 GPU 的计算吞吐，它划分的密集块通常比较大（比如 $l’=32$），且通常不重叠。

当你要评估细粒度选择分支的第 $j$ 个大块（大小是 32）重要不重要时，你手里只有一堆压缩分支的小 Token（每个代表 8 个空间）的注意力得分。
因为网格大小不一样，你没法直接连线。你必须把物理空间上“落在这个 32 维大块内部”的所有小压缩 Token 的得分给挑出来，全部加在一起。

其中$\mathbf{p}_t^{\text{slc}}[j]$表示第t个token与第j个细粒度块之间的注意力分数，$\mathbf{p}_t^{\text{slc}}$则表示这个token与所有细粒度块的注意力得分，其大小为$1×m$，其中m为细粒度块数量

举例：
压缩步长 $d=8$，压缩窗长 $l=16$（有 50% 重叠）；选择块大小 $l’=32$，计算细粒度选择分支第 $j=10$ 个大块的得分 $\mathbf{p}_t^{\text{slc}}[10]$，第10个大块对应的token压缩序列应该是288-320，那么对应的粗粒度块应该是36-40

根据计算，公式最终把第 36, 37, 38, 39, 40 这几个压缩 Token 的注意力得分全部加了起来。

3、多头注意力得分统一

中心思想：对于多头attention，每个头都会对应一个独立的$\mathbf{p}^{\text{slc}}$(大小为$T×m$)，那么需要将多头中的每个头的注意力得分综合起来，即求和

如果每个头得到的Top-K个细粒度块都不一样，假设有8个head，因为这 8 个 Head 共享同一份 KV Cache 内存。当 GPU 的线程束（Warps）准备去显存里搬运数据时，它发现这 8 个 Head 要去 8 个完全不连续的、甚至隔了十万八千里的内存地址（第 3、50、99 块）去读数据。

这在显存里叫非合并访问（Uncoalesced Memory Access）/ 内存分叉（Memory Divergence）。
GPU 无法一次性大批量把数据运回来，只能像送外卖一样，派 8 个小哥分别去 8 个地方单点取货。稀疏注意力本来是为了省带宽、提速，结果多头的意见不合，反而把 GPU 堵死在频繁读取非连续显存的延迟（Latency）里了。

  • $H$：这个 KV 共享组里的 Head 总数（比如 8 个）。
  • $\mathbf{p}_t^{\text{slc}, (h)}$：第 $h$ 个 Head 独立算出来的、长为 $m$ 的细粒度大块得分向量。
  • $\mathbf{p}_t^{\text{slc}’}$：强行求和后、全组共享的“终极总榜单”。

所以初始的$\mathbf{p}_t^{\text{slc}’}$大小为$1×m$，H个头则$H×1×m$，$\mathbf{p}_t^{\text{slc}’}$的大小为$1×m$，即将所有头的注意力得分进行了统一

  • $n$：代表被你挑中的优胜大块的数量（Top-$n$）。
  • $l’$：每个细粒度大块内部包含的原始 Token 的个数。
  • $n \cdot l’$：这些块被硬性打捞上来后，首尾相连地拼接在一起。比如挑出了 $n=4$ 个大块，每个块含有 $l’=16$ 个 Token，那么这里拼出来的矩阵长度就是 $4 \times 16 = 64$ 个原始 Token Key。

前面粗粒度细粒度饶了一大圈，最终还是选择的原始token进行注意力分数的计算，原因就是需要通过前面的这些块快速计算哪些token与当前的token的关联性更高，获取后再拿原始的token去算注意力得分

3、Sliding Window

在 DeepSeek NSA 论文的体系中，当前时间步 $t$ 的 Query 向量 $\mathbf{q}t$ 能够全量、无损交互的局部窗口 Key 集合定义为：$$\tilde{K}_t^{\text{local}} = f_K^{\text{local}}(\mathbf{k}{:t}) = \left{ \mathbf{k}_i \;\middle|\; \max(1, \, t - l_w + 1) \le i \le t \right}$$

其实就是获取当前token的前n个token，这个n是一个固定值，然后对这些token计算完整的注意力

4、门控加权融合（Gated Mixture / Gated Fusion）

对于某个token，现在有选出来了三部分其他tokens与其对应：
| 分支名称 | 调阅的 Key 矩阵符号 | 向量类型 | 覆盖的原始文本范围 | 核心工程贡献 |
| :—- | :—- | :—- | :—- | :—- |
| 滑动局部窗口
(Sliding Window) | $K_w$ / $\tilde{K}_t^{\text{local}}$ | 原始密集 Token
（无损） | 临近的最近 $l_w$ 个词
(如 $9489 \sim 10000$) | 保底语法结构：
确保近距离语义和语法的绝对流畅。 |
| 动态细粒度选择
(Token Selection) | $\tilde{K}_t^{\text{slc}}$ | 原始密集 Token
（无损） | 由公式(9)(10)(11)精选出的
$n$ 个远端密集大块 | 大海捞针（针）：
确保历史深处的微观事实（人名、数字）精准命中。 |
| 粗粒度压缩块
(Compressed Blocks) | $\tilde{K}_t^{\text{cmp}}$ | 滑动压缩 Token
（模糊/脱水） | 远端窗口外所有的历史块
(如 $1 \sim 9488$) | 全局大局观（海）：
确保模型不遗忘宏观的上下文背景和核心主旨。 |

融合的核心工程手段：门控加权融合（Gated Mixture / Gated Fusion）：

  1. 三大分支独立计算（各自交卷）
    对于当前的第 $t$ 个 Token 的 $\mathbf{q}_t$，它要平行地跑三遍经典的注意力计算：
  • $\text{Attn}(\mathbf{q}_t, \tilde{K}_t^{\text{win}}, \tilde{V}_t^{\text{win}})$：去跟近处的原始滑动窗口算，吐出一个输出向量 $\mathbf{o}_t^{\text{win}}$。
  • $\text{Attn}(\mathbf{q}_t, \tilde{K}_t^{\text{slc}}, \tilde{V}_t^{\text{slc}})$：去跟打捞出来的远端细粒度密集块算，吐出一个输出向量 $\mathbf{o}_t^{\text{slc}}$。
  • $\text{Attn}(\mathbf{q}_t, \tilde{K}_t^{\text{cmp}}, \tilde{V}_t^{\text{cmp}})$：去跟全量的远端粗粒度压缩代表算，吐出一个输出向量 $\mathbf{o}_t^{\text{cmp}}$。

这三个输出向量的物理形状完全一致，都是 $1 \times d_v$（单个 Head 视角下）。

  1. 门控系数（$g_t^c$）的动态微调
    论文中提到：
    $g_t^c \in [0, 1]$ is the gate score … derived from input features via an MLP and sigmoid activation.
    这意味着，模型配了一个极轻量的小动态网络（MLP + Sigmoid）。它会根据当前 Token 自身的上下文特征，动态地实时算出三个 0 到 1 之间的权重系数（$g_t^{\text{win}}, g_t^{\text{slc}}, g_t^{\text{cmp}}$）。

其中$\mathbf{x}_t$ 为 当前Token 的输入特征（Input Features）
