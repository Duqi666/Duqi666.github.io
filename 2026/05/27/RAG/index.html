Pre-Retrieval（检索前优化）

1. Query Rewriting / Expansion (查询改写与扩展)

A. 拓展查询（Query Expansion / Multi-Query）

  • 原理： 用户的一种提问方式可能无法命中向量空间的最佳区域。我们可以利用一个轻量级大模型（或微调后的小模型），让它基于原始 Query 生成 NNN 个语义相近但措辞不同的候选问题（例如从不同角度、使用不同的术语）。
    • System Prompt 示例：你是一个专业的搜索优化助手。请将用户输入的原始查询，扩展为 4 个语义相近、但使用更专业技术术语、或从不同角度切入的搜索关键词/句子。直接返回 JSON 数组，不要任何解释。
  • 工程落地： 将这 N+1N+1N+1 个 Query 并发进行向量检索，最后通过 RRF（Reciprocal Rank Fusion） 算法对所有检索回来的文档进行重排去重。这能显著提升检索的召回率（Recall）。

结果融合（RRF, Reciprocal Rank Fusion）

由于不同 Query 检索出来的文档列表会有重合，且分数不可直视对比，业界标准做法是用 RRF 算法 进行重新排序和去重。

RRF 的核心思想是：不看向量绝对得分，只看文档在各个检索结果里的“排名（Rank）”。如果一个文档在 5 次检索中都排在前三名，那么它的综合权重就会变得极高。

RRF_Score(d)=∑q∈Q1k+rq(d)RRF\_Score(d) = \sum_{q \in Q} \frac{1}{k + r_q(d)} RRF_Score(d)=q∈Q∑​k+rq​(d)1​

(其中 rq(d)r_q(d)rq​(d) 是文档 ddd 在查询 qqq 的检索结果中的排名，kkk 通常取常数 60)

RRF的优势：

  • 倾向于奖励“高共识”的文档： 如果一个文档被多个不同的改写查询共同看中（即使排名不是第一），RRF 会认为这个文档的鲁棒性极高，更契合用户真正的底层意图。
    • 换句话说就是倾向于选择多个query选择的文档，而非一个query选择的高分文档
  • 抹平了不同模型的物理分值差异： 假设 Query 1 用的是密集检索（向量库算出的 Cosine 相似度是 0.85），Query 2 伴随了 BM25 检索（算出的 BM25 Score 是 18.5）。这两种分值根本没法直接加减。而 RRF 只提取它们在各自队列里的 Rank（第几名），天然完成了归一化（Normalization）。
    • 或者，在某种情况下文档A在第一个query的查询中获得了极高的分数，但是在其他query的查询中都未存在，而文档B在多个query中都存在，但是分数不高，这种情况下需要选择B而非A，使用排名可以很好的完成这一功能

B. 假设性文档正文（HyDE: Hypothetical Document Embeddings）

  • 原理： Query 与 Doc 在向量空间中往往是不对齐的（问题通常很短，答案通常很长）。HyDE 的核心思想是将问题直接丢给模型生成一份答案，然后拿这个答案作为新的Query去做向量检索。
  • 步骤：
    • 让 LLM 接收原始 Query，并生成一篇假设性的、可能错误的但格式符合标准答案的文档（Hypothetical Document）。
    • 不对这篇文档做真伪校验，直接将这篇“假答案”送入 Embedding 模型转化成向量。
    • 用这个“假答案”的向量去数据库中检索真实的 Chunks。
  • 为什么有效： 在向量空间中，“答案向量 →\rightarrow→ 答案向量”的距离，通常比“问题向量 →\rightarrow→ 答案向量”的距离更近、更稳定。即便生成的答案有一些错误，但是其在语义上已经非常丰富了，完全可以在数据库中找到全面且正确的文档

⚖️ HyDE 的优缺点与边界

优点：

  • 完美解决“零发散”提问： 用户提问可能只有几个字，而 HyDE 能强行把语义空间撑大，补充大量背景相关的专有名词，从而极大提升了稠密检索的 Recall（召回率）。
  • 不依赖特定微调： HyDE 是一种 Zero-shot 的全链路范式。你不需要花大代价去用对比学习训练自己的 Embedding 模型，直接用现成的 LLM + 现成的 Embedding，就能在特定垂直领域（如医疗、法律）拉升一截效果。

缺点与踩坑点（工业界痛点）：

  • 延迟（Latency）增加： 这是最致命的。RAG 检索原本只需要几十毫秒，现在检索前必须先让大模型生成一段话。即使只生成 150 个 Token，在没有流式输出的情况下也会增加 300ms ~ 500ms 的延迟。
  • 严重误导（Catastrophic Misleading）： 如果用户的提问涉及非常罕见、极其小众、或者完全反常识的知识，LLM 可能会写出一篇南辕北辙的假文档。
    • 举个极端例子： 用户问“为什么明朝的程序员喜欢用 Python？”。LLM 可能会顺着胡编一篇关于明朝程序员用 Python 的假论文，结果导致向量库检索出来的全是明朝历史或 Python 历史，把真正需要的“查无此人/报错提示”给淹没了。

🛠️ HyDE 加强套路

为了在实际生产中落地 HyDE，算法工程师通常会对其进行改造：

  • Multi-HyDE（多视角生成）：通过设置 temperature=0.7 或改变 Prompt 视角，让 LLM 一口气生成 MMM 篇短小的假设性文档。然后将这 MMM 个向量做平均池化（Mean Pooling），或者各自检索后用 RRF 融合。这能有效对冲单次生成遭遇严重幻觉的风险。
  • 条件触发 HyDE（Selective HyDE）：引入轻量级的意图识别。如果 Query 是标准的关键词（如“Linux 4.19 内核编译报错 137”），直接走传统检索；如果 Query 是开放式的、概念性的（如“如何评价基于体素的 RSRP 预测算法的优缺点”），再触发 HyDE。

2. Query Decomposition (查询拆解)

1. 并行拆解：Sub-Query Decomposition (子查询拆解)

当用户的提问是并列结构、各子问题之间没有依赖关系时，适用此范式。

算法流

  • Planner（规划器）： 利用小模型作为 Planner，将原始 Query 拆解为 NNN 个可以独立并行检索的子 Query。
  • Execution（执行）： 触发 NNN 路并发检索，各自召回对应的 Chunks。
  • Synthesis（融合）： 将 NNN 组检索结果同时喂给 Generator（生成器），让其进行横向对比和综合回答。

工业界经典实现

  • LlamaIndex 中的 SubQuestionQueryEngine。

2. 串行/递进拆解：Multi-Step / Interleaved Retrieval (多步/交织检索)

当用户的提问是深度依赖结构（即回答问题 B 必须先知道问题 A 的答案）时，适用此范式。这种方式更偏向于 Agentic RAG（智能体化RAG）。

算法流

  • Step 1： Planner 分析 Query，发现无法一步到位，于是提取出“前置子问题 1”。
  • Step 2： 针对“前置子问题 1”进行检索，并让 LLM 生成一个阶段性中间答案（Intermediate Answer）。
  • Step 3： 结合原始 Query 和这个中间答案，生成“子问题 2”，再次触发检索。
  • Step 4： 循环往复，直到模型判断信息足够，生成最终答案。

工业界经典实现

  • IRCoT (Interleaved Retrieval with Chain-of-Thought) 和 FLARE (Forward-Looking Active Retrieval)。模型在写 CoT（思维链）的过程中，一旦发现接下来要写的句子缺乏事实依据，就临时停下来触发一次检索，把检索结果补进上下文后继续往下写。

举例

3D物理特征与无线通信建模 为了更直观，我们来看一个复杂的跨学科技术支持场景：
  • 用户的原始输入：

    “如果我们要处理 100GB 的 3D 物理地图 CSV 文件，并提取 12 维空间特征来进行 RSRP 预测，目前基于体素（Voxel）的方法和基于点云（Point Cloud）的方法相比，哪种在 GPU 上的算力开销更小？我们的初版模型要在 6 月 30 日交付，哪种方案风险更低？”

直接拿这段话去向量数据库检索，大概率会召回一些谈论“自动驾驶点云”或“无线电波传播基础理论”的宽泛文档，无法切中用户的特定工程痛点。

🛠️ 拆解器（Planner）的拆解输出
小模型（如 Qwen-7B-Instruct）接收到该 Query 后，会将其重构为一张任务网络：

                          ┌──> Sub-Query 1: "3D 空间 Voxel  voxelization GPU 算力消耗 RSRP 预测"
                          │
[原始复杂 Query] ──> Planner ──┼──> Sub-Query 2: "点云 Point Cloud 特征提取 GPU 显存 算力开销"
                          │
                          └──> Sub-Query 3 (依赖前两路结果): "大规模 3D 地图 CSV 12维特征 数据管线 落地风险/成熟度"
  1. 子查询 1 (并行)： “3D 空间体素化（Voxelization）算法在 GPU 上的算力开销与 RSRP 预测性能评估”
  2. 子查询 2 (并行)： “基于点云（Point Cloud）的 3D 空间物理特征提取算法的 GPU 显存与计算复杂度分析”
  3. 子查询 3 (依赖前两路检索结果的合成)： “处理百 GB 级别 3D 空间地图 CSV 数据时，Voxel 与 Point Cloud 方法的数据管线（Data Pipeline）开发周期与交付风险对比”
